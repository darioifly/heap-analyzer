| Metric | Pre-F2.S10 (manual flow, morphological DTM) | Post-F2.S10 (DJI flow, ground classification) |
|---|---|---|
| Heaps detected | 21 (with rail/road false positives) | 16 |
| Total volume | 5 084 m³ | 16 760 m³ |
| Pipeline duration | 516 s | 304 s |
| DTM method | `morphological` (flat, base=213.79 m) | `ground_classification` (locally varying) |
| DTM confidence | 0.90 | 0.95 |

The +230% volume swing is not a regression — it reflects the morphological
DTM previously clipping heap tops (the kernel of 50 px = 5 m cannot "see
through" scrap piles 20+ m wide, so the opened raster treated heap tops as
ground, collapsing heights and volumes).

## Key design decisions:

### 1. Prefer `map/dsm.tif` over `models/pc/0/terra_dem/dem.tif`
The `map/` DSM is the one DJI Terra exports from its 2D pipeline after the
"2dPipeline_done" sentinel; `terra_dem/dem.tif` is the intermediate from the
3D reconstruction step and has lower quality at edges. The scanner falls
back to `terra_dem/dem.tif` with a warning only when `map/dsm.tif` is
absent.

### 2. `task.json` is intentionally ignored
The prompt assumed `task.json` was plain JSON, but on the real DJI output
it is a hex-encoded encrypted blob. Rather than reverse-engineer the format,
we derive CRS from `.prj` sidecars (WKT, easy to parse with pyproj) and
survey date from file mtime (editable in the UI).

### 3. Classification sampled on first 100k points only
`cloud_merged.las` can be 1-2 GB with 60M+ points; a full scan to probe for
class=2 is wasted I/O. We sample the first chunk via `laspy.chunk_iterator`.
On the real sample this gives 92.8% class=2 coverage in < 1s.

### 4. Ground-classification DTM: nearest-neighbour Voronoi fill
After rasterising class=2 points onto the DSM grid (min Z per cell), empty
cells are filled with the nearest populated cell's value via
`scipy.ndimage.distance_transform_edt(..., return_indices=True)`. This is
O(N) in the grid size and produces no interpolation artefacts. Coverage
threshold: 5% (below → fall back to morphological).

### 5. Files referenced by default (not copied)
Drone imagery folders are multi-GB; forcing every import to copy them would
waste disk. The dialog defaults `copyFiles` to OFF. A clear warning tells
the user that moving the DJI folder later will orphan the survey.

### 6. Precomputed DSM threaded via `processing_params`
Instead of adding a dedicated DB column for the DJI DSM path, we JSON-
serialise `{precomputed_dsm_path: <path>}` into the existing
`processing_params` column. The pipeline reads it via
`ProcessingConfig.precomputed_dsm_path`. This keeps the DB schema lean.

## New IPC channels:
- `dji:scanFolder` — spawns Python `scan-dji-terra` and returns a typed
  manifest (or `{ok:false, code, message}` on failure)
- `dji:importSurvey` — creates a survey row with `source_type='dji_terra'`
  and (optionally) copies files into a timestamped sibling directory

## DB schema changes:
- `surveys.source_type TEXT DEFAULT 'manual'` — `'manual' | 'dji_terra'`
- `surveys.dji_folder_path TEXT` — original DJI root path (null for manual)
- Idempotent migration: existing rows get the default `'manual'` value from
  the `ALTER TABLE ... DEFAULT` clause, no data migration needed.

## New Python CLI commands:
- `heap-analyzer scan-dji-terra --folder <path>` — emits manifest JSON Lines

## Known limitations:
- Existing vitest DB test files (db.test.ts, migrations.test.ts) cannot run
  in the root vitest config due to the pre-existing NODE_MODULE_VERSION
  mismatch between better-sqlite3 (compiled for Electron ABI) and the root
  Node runner. Same infra issue documented in F7 report.
- ImportDJIDialog.test.tsx requires the frontend-workspace vitest config
  (happy-dom + jest-dom setup). Same situation as existing
  SurveyList.test.tsx.
- File copy on import is blocking (no progress events). Acceptable for v1;
  the file sizes are predictable (~1-2 GB for LAS, 100-300 MB for TIFF).
